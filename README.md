# 🚗 Цифровая приборная панель Lada Vesta (Qt6/QML + ESP32 CAN Gateway)

Цифровая приборная панель на **C++ / Qt6 / QML**, получающая данные с автомобильной
шины **CAN** через микроконтроллер **ESP32**. Поддерживает два режима:

- **Локальная симуляция** — физика разгона/торможения считается прямо в Qt (`CarController`),
  без всякого железа, управление с клавиатуры (W/A/S/D + Tab для переключения режима).
- **Внешние данные (ESP)** — приборка получает и отображает данные, пришедшие с ESP32.

Переключение между режимами — клавиша **Tab** (`useExternalData` в QML).

---

## 📍 Текущий статус проекта (эта версия)

Сейчас проект в состоянии **тестового стенда на одной плате**: один ESP32
одновременно и **генерирует** тестовые CAN-кадры программно (без реальной шины
и без CAN-трансивера), и **декодирует** их через `CanDecoder`, и **отправляет**
готовый JSON в Qt по **USB Serial** (COM5, 115200 бод).

Это осознанный промежуточный шаг — так удобно проверять весь путь
`генерация → декодирование → парсинг → QML`, не имея пока второй платы и
физической CAN-шины.

```text
┌─────────────────────────────┐
│      ESP32 (одна плата)     │
│  fakeCAN() → CanDecoder →   │
│  JSON → Serial (COM5)       │
└──────────────┬──────────────┘
               │ USB Serial, 115200 бод
┌──────────────┴───────────────┐
│         Qt6 / QML UI         │
│  CanReceiver → Q_PROPERTY →  │
│         приборка             │
└──────────────────────────────┘
```

## 🎯 Целевая архитектура (план — две платы ESP32)

Когда появится вторая плата, работа делится на два независимых узла:

**ESP-A — «Симулятор машины»**
Программно генерирует реалистичные CAN-кадры (скорость, обороты, передача,
пробег, лампы и т.д.) и отправляет их **по-настоящему**, по физической
CAN-шине через CAN-трансивер (SN65HVD230 / MCP2551), 500 кбит/с,
11-битные ID. Заменяет собой сегодняшний `fakeCAN()`.

**ESP-B — «Шлюз/приёмник»**
Слушает реальную CAN-шину через свой трансивер, декодирует кадры тем же
`CanDecoder` + `can_signals.json`, но вместо USB Serial отправляет готовый
JSON **по Wi-Fi** (TCP или UDP) в Qt-приложение.

**Qt/QML** — меняется только транспорт: `CanReceiver` вместо
`CreateFileA`/COM-порта открывает `QTcpSocket`/`QUdpSocket`. Логика парсинга
JSON и биндинги в QML остаются те же.

```text
┌─────────────┐   CAN H/L, 500 kbps   ┌─────────────┐   Wi-Fi (TCP/UDP)   ┌──────────────┐
│   ESP-A      │ ───────────────────► │   ESP-B      │ ───────────────────► │  Qt6 / QML    │
│  Симулятор   │   через трансивер     │  Шлюз/приём  │      JSON            │   приборка     │
└─────────────┘                        └─────────────┘                       └──────────────┘
```

### Распределение задач по платам

| Задача | Плата |
|---|---|
| Генерация сигналов (симуляция физики машины) и отправка в CAN | ESP-A |
| Приём CAN-кадров, декодирование, отправка JSON по Wi-Fi | ESP-B |
| Приём JSON, обновление `Q_PROPERTY`, реакция QML на сигналы | Qt/QML (без изменений в этой части) |

### Что нужно сделать для перехода

1. Взять код `fakeCAN()` из `main.cpp` — перенести генерацию кадров на ESP-A,
   заменить `canDecoder.decodeFrame(frame)` на реальную отправку в CAN
   (`CAN.beginPacket()/write()/endPacket()` из библиотеки `sandeepmistry/CAN`,
   которая уже в `platformio.ini`).
2. На ESP-B оставить `CanDecoder` + `can_signals.json` как есть, но принимать
   кадры через `CAN.parsePacket()` вместо программной генерации.
3. На ESP-B заменить `Serial.print(...)` в `sendDashboard()` на отправку по
   Wi-Fi (`WiFiClient`/`AsyncTCP` или `WiFiUDP`).
4. В Qt — переписать `CanReceiver::listenPort()`: убрать Win32 `CreateFileA`
   на COM-порт, подключить `QTcpSocket`/`QUdpSocket`, слушающий IP/порт ESP-B.
   Формат JSON и вся остальная логика `processJson()` не меняются.
5. Собрать физический стенд: ESP-A и ESP-B соединить проводами CAN H/L через
   трансиверы, проверить `candump` (см. §6 `CAN_ПРИВЯЗКА_ИНДИКАТОРОВ.md`), что
   кадры реально идут по шине и совпадают с тем, что было в программной
   симуляции.

## 📡 Документ «как формируется каждый сигнал» (текущая симуляция)

Ниже — таблица того, что прямо сейчас считает `fakeCAN()` в `main.cpp`.
Это и есть текущая модель поведения машины; при переносе на ESP-A эта же
логика должна быть сохранена без изменений (либо расширена).

| Сигнал (`target`) | CAN ID | Как считается сейчас | Единицы |
|---|---|---|---|
| `VehicleSpeed` (speed) | `0x29A` | `60 + 50·sin(millis()/4000)` — плавные колебания ~10..110 км/ч | км/ч |
| `currentGear` (gear) | `0x421` | Ступенчато по диапазону скорости: `<20`→1, `<40`→2, `<65`→3, `<90`→4, иначе→5; `<1`→D | — |
| `EngineRPM` (rpm) | `0x186` | `900 + rpmFraction · 5100`, где `rpmFraction` — доля пути скорости внутри диапазона текущей передачи | об/мин |
| `DistanceTotalizer` (odometer) | `0x5D7` | Интеграл скорости по времени: `odometerKm += speedKmh · Δt_часы`, стартовое значение 12345.60 км | км |
| `EngineCoolantTemp` (coolant) | `0x5DA` | Фиксировано 90 °C (raw=130, offset −40) | °C |
| `fuelLevel` (fuel) | `0x5E5` | Пилообразно 0↔100%, шаг ±0.5% за тик | % |
| `highBeam` / `lowBeam` | `0x481` | Переключаются каждые 3 сек (`millis()/3000 % 2`) | флаг |
| `turnLeft` / `turnRight` | `0x481` | Мигают синхронно каждые 0.5 сек (`millis()/500 % 2`) | флаг |
| `position` (габариты) | `0x60D` | Переключается каждые 4 сек | флаг |
| `fogFront` / `fogRear` | `0x60D` | Переключаются каждые 4.5 / 5 сек | флаг |
| `checkEngine`, `oilPressure`, `batteryWarning`, `absWarning`, `steeringWarning`, `escWarning`, `doorsWarning` | `0x625` | Каждая лампа мигает со своим периодом (2.0–5.0 сек), независимо друг от друга | флаг |
| `seatbelt` | `0x498` | Переключается каждые 1.5 сек | флаг |
| `HandBrakeSwitch` (handBrake) | `0x4F8` | Переключается каждые 2 сек, значение 2=затянут/1=отпущен | значение (2 бита) |

> ⚠️ ID кадров `0x481`, `0x498`, `0x625`, `0x4F8`, `0x60D` частично реверс-инжинерены
> сообществом (Drive2) и **не подтверждены `candump`** на реальной машине —
> см. предупреждение и методику проверки в `CAN_ПРИВЯЗКА_ИНДИКАТОРОВ.md` (§6).
> `VehicleSpeed`, `EngineRPM`, `EngineCoolantTemp`, `DistanceTotalizer` — из
> Renault RNr DBC-выгрузки (`can_signals.json`), считаются более надёжными,
> но тоже требуют проверки под конкретный год/комплектацию машины.

### Ключевые файлы

| Файл | Назначение |
|---|---|
| `main.cpp` (ESP32) | Точка входа прошивки, `fakeCAN()` — генерация тестовых кадров, `syncDashboardSignals()`/`sendDashboard()` — сборка и отправка JSON |
| `CanDecoder.h/.cpp` | Универсальный декодер: читает `can_signals.json`, парсит кадры по картам `value`/`flag`, отдаёт значения по имени `target` |
| `CanTypes.h` | Структуры `CanFrame`, `CanSignalDef`, `SignalType` |
| `can_signals.json` | Карта CAN-сигналов: ID → список `target` с байтами/битами/factor/offset. Renault RNr DBC-выгрузка + добавленные вручную блоки под индикаторы Весты (`0x481`, `0x498`, `0x4F8`, `0x5E5`, `0x60D`, `0x625`) |
| `CAN_ПРИВЯЗКА_ИНДИКАТОРОВ__2_.md` | Матрица привязки ламп/показаний к блокам-источникам на реальной машине, методика проверки `candump` |
| `CANматрица_Lada_Vesta.pdf` | Справочная CAN-матрица (Renault RNr) с байт/бит раскладкой сигналов |
| `canreceiver.h/.cpp` (Qt) | Приём данных от ESP32 (сейчас — Serial/COM5), парсинг JSON, `Q_PROPERTY` для QML |
| `carcontroller.h/.cpp` (Qt) | Локальная симуляция физики машины для режима без ESP (клавиатура) |
| `main_qml.txt` | UI приборки: тахометр, спидометр, топливо/температура, иконки предупреждений, OBC-дисплей |

## 📦 Формат обмена данными (ESP32 → Qt)

Сейчас — построчный JSON по USB Serial (одна строка = один пакет, `\n`-терминированный):

```json
{"rpm":2450.00,"speed":78.30,"coolant":90.00,"fuel":42.50,"odometer":12389.14,"gear":"3","highBeam":0,"lowBeam":1,"turnLeft":0,"turnRight":0,"checkEngine":0,"oilPressure":0,"batteryWarning":1,"absWarning":0,"steeringWarning":0,"escWarning":0,"doorsWarning":0,"seatbelt":1,"handBrake":0,"position":1,"fogFront":0,"fogRear":0}
```

В целевой версии формат JSON остаётся тем же, меняется только транспорт
(Wi-Fi TCP/UDP вместо Serial).

## 🛠 Требования к системе

### Qt-приложение (интерфейс)
- **Qt SDK 6.x** (Qt Quick, QML)
- **CMake** ≥ 3.16 + Ninja/Make
- MSVC (Windows) / GCC (Linux) / Clang (macOS)
- Qt Creator (рекомендуется), VS Code или CLion

### Прошивка ESP32
- Плата **ESP32 DevKit** (сейчас одна, в целевой версии — две)
- Для целевой версии: CAN-трансивер **SN65HVD230** (рекомендуется, 3.3V) или
  **TJA1050** (нужен преобразователь уровней) на каждую плату
- **PlatformIO** (см. `platformio.ini`) или Arduino IDE 2.x
- Библиотеки: `bblanchon/ArduinoJson @ ^7.0.0`, `sandeepmistry/CAN @ ^0.3.1`

## 🚀 Запуск (текущая версия, один ESP32)

### 1. Прошивка ESP32
```bash
# В папке проекта прошивки
pio run -t upload           # залить код
pio run -t uploadfs         # залить can_signals.json в LittleFS (ОБЯЗАТЕЛЬНО
                             # при любом изменении can_signals.json!)
pio device monitor -b 115200
```
В Serial Monitor должно появиться:
```
[CAN] Карта сигналов загружена: N сообщений
CAN MAP OK
{"rpm":...,"speed":...,...}
```
Если видите `[CAN] Ошибка разбора JSON` — в `can_signals.json` где-то пропущена
или лишняя запятая; проверьте через `python3 -m json.tool can_signals.json`.

### 2. Запуск Qt-приложения
```bash
mkdir build && cd build
cmake ..
cmake --build .
./dashboard        # Linux/macOS
dashboard.exe      # Windows
```

⚠️ Serial Monitor (VS Code/PlatformIO) и Qt-приложение **не могут одновременно**
держать открытым один COM-порт — закройте монитор перед запуском Qt.

### 3. Проверка
- **Tab** — переключение между локальной симуляцией (`CarController`) и
  данными с ESP32 (`CanReceiver`).
- В режиме ESP все иконки и стрелки должны реагировать на данные автоматически,
  без нажатия клавиш (кроме `Key_1`, который остаётся демо-переключателем
  только для локального режима).
